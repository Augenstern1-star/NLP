# 《地书翻译机》技术报告草稿

## 1. 作品概述

《地书翻译机》是一个离线网页作品。用户输入中文短句后，系统根据已有《地书》图形素材与标注数据，匹配出相近的图形或图形组，并展示字面意义、语境意义、可能误读和翻译置信度。

作品核心问题是：通用图形语言能否稳定地承载自然语言意义？系统不是追求唯一正确翻译，而是把翻译过程中的不确定性展示出来。

## 2. 数据来源

- 《地书》图形素材：`地书标注系统 V1.0/images/auto_cut_segments/`
- 《地书》图形标注数据：`《地书》标注数据/`
- 中文白话版标注数据：`《地书》白话版标注数据/`
- 自然语言侧规范：`数据标注规范.md`

系统使用的主要标注任务包括：

- `literal_gloss`
- `pragmatic_meaning`
- `free_translation`
- `event_description`
- `semantic_primitives`
- `pos_like_category`
- `morphological_features`

## 3. 数据清洗流程

清洗脚本为 `scripts/build_translator_data.py`，主要步骤如下：

1. 读取《地书》标注导出文件，统一为 JSONL 行格式。
2. 读取白话版中文标注数据，兼容 `unit_id + task字段` 的非标准格式。
3. 按 `数据标注规范.md` 统一字段：`target_id`、`target_type`、`task_id`、`value`、`source_file`。
4. 从字面释义、语境释义、自由翻译和事件描述中抽取候选概念。
5. 将词性、形态、语义基元保留为标签，不直接作为翻译概念。
6. 生成作品运行所需的 `concept_index.json` 和 `examples.json`。

生成结果：

- 《地书》标注行：46,494
- 白话文本标注行：31,181
- 候选概念：13,522
- 作品概念索引：260
- 精选演示句：10
- LLM 批次：18

## 4. 大模型使用方式

本作品采用“制作阶段使用大模型、运行阶段不调用大模型”的方式。

代码先把原始数据压缩成概念候选和 LLM 批次文件，位于：

```text
data/translator/llm_batches/
```

这些批次可以交给大模型进行概念合并、解释润色和误读生成。网页运行时只读取已经生成好的本地数据，因此课堂展示不依赖网络、API key 或模型服务。

网页还提供“AI 补译”交互：当本地匹配出现不可译空位时，系统会把空位、原句和候选概念压缩成 JSON payload。本地 `server.js` 后台代理可以调用 DeepSeek 或其他 OpenAI-compatible API，自动选择已有 `concept_id` 并回填到图形序列中。API key 只用于本次本地请求，不写入作品数据文件。网页也保留手动 prompt 回填方式，便于无后台环境下演示。

## 5. 网页实现

网页位于：

```text
dishu-translator/index.html
```

主要模块：

- 输入区：输入中文短句或选择示例句。
- AI 补译区：填写 API endpoint、model、API key 后，由本地后台自动补足不可译空位；也支持手动 prompt 回填。
- 地书输出区：横向展示匹配到的图形序列。
- 解释区：展示字面意义、语境意义、可能误读、来源 ID。
- 置信度：根据候选概念和图形匹配强度给出估计值。

自由输入采用本地最长同义词匹配。无法匹配的片段显示为“不可译空位”。

## 6. 测试结果

已完成以下检查：

- 清洗脚本能读完整个数据集并生成结果。
- `app.js` 通过 Node.js 语法检查。
- `build_translator_data.py` 通过 Python 编译检查。
- 10 条示例句中的概念 ID 均存在。
- 260 个概念的主候选图片路径均存在。
- 网页不依赖运行时 API。

## 7. 局限与改进

当前版本的概念合并主要依赖规则，仍可能存在同义词没有完全合并、部分概念过宽或过窄的问题。后续可以使用 `llm_batches` 进一步让大模型合并同义概念、筛掉弱候选，并为每个概念生成更稳定的展览文本。
